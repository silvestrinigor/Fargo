- Avoid using real or personal names (e.g., Igor) as test values. Use neutral and generic data instead.

- Unit test classes should follow the naming convention: <ClassName>Tests.
Example: tests for Article → ArticleTests.

- When appropriate, create a dedicated faker/builder class to generate test data.

    - Place the faker class below the test class.

    - Use the naming convention: <ClassName>Fakers.
    Example: ArticleFakers.

- When testing a class that depends on another class, reuse the corresponding faker class instead of duplicating setup logic.

- Test methods should follow the pattern:
MethodName_Should_ExpectedBehavior_When_Condition
    - Example: Create_Should_ReturnArticle_When_DataIsValid

- Keep each test focused on one behavior only (single responsibility).

- Avoid overly long test methods—prefer readability over compactness.

- Always structure tests using:

    - Arrange – setup data

    - Act – execute behavior

    - Assert – verify results

- Do not mix multiple asserts for unrelated behaviors in the same test.

- Tests must be independent (no shared state between tests).

- Avoid hardcoded values when possible—prefer faker-generated data.

- Use meaningful values when testing edge cases (e.g., empty string, null, max length).

- Always include tests for:

    - Null inputs

    - Invalid inputs

    - Boundary values (min/max limits)

- Validate both success paths and failure paths (exceptions).
